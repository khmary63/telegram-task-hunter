
# Парсер Telegram-чатов для поиска лидов

## Что строим

Два связанных компонента:

1. **Воркер на VPS (Python + Telethon)** — слушает Telegram, фильтрует сообщения по ключевикам, пишет совпадения в базу.
2. **Веб-интерфейс в Lovable (TanStack Start + Lovable Cloud)** — управление ключевыми словами, чатами, нишами, фильтрация по дате, просмотр найденных лидов с никами и ссылками на сообщения.

База одна на оба компонента — Lovable Cloud (Postgres). Воркер пишет туда через service_role ключ, UI читает через RLS.

---

## Рекомендация по доступу (вы спросили совет)

Использовать **личный Telegram-аккаунт через MTProto (Telethon)**. Bot API не подходит — бот видит только чаты, где он админ, а вам нужны чужие публичные группы.

Важные предупреждения:
- Заведите **отдельный аккаунт** на отдельную SIM — не основной. При агрессивном чтении Telegram может временно ограничить или забанить аккаунт.
- Не вступайте в сотни чатов разом. Telethon-сессия должна "прогреться" — добавляйте чаты постепенно (5–10 в день первую неделю).
- Не отправляйте сообщения с этого аккаунта автоматически — только чтение.
- `api_id` / `api_hash` получаете на my.telegram.org, авторизация по SMS-коду один раз при первом запуске воркера.

## Про "автопоиск чатов" — честное ограничение

Telegram **не даёт официального API для поиска групп по теме**. Варианты:

- **tgstat.com** — есть платный API (~₽2–5к/мес), позволяет искать каналы/чаты по ключевикам и категориям. Это самый чистый путь к автопоиску.
- **Бесплатно**: воркер дополнительно собирает из сообщений ссылки вида `t.me/...` и `@username`, складывает в таблицу `chat_suggestions`. В UI вы их видите и одной кнопкой добавляете в мониторинг. Так список органически растёт от уже подключённых чатов.

**Предлагаю стартовать с гибрида:** ручной seed + автосбор suggestions из уже подключённых чатов. Подключение tgstat API — опционально на втором этапе, если первого мало.

## Логика фильтрации

Для каждого нового сообщения воркер проверяет:

1. **Точные ключевые фразы** (ваш список «хочу ИИ попробовать», «нужна CRM», ...) — нечувствительно к регистру, с нормализацией пробелов/пунктуации.
2. **Семантическое совпадение с нишами IEEE** (внедрение ИИ, боты, веб/мобильные, вайб-кодинг, лендинги, LLM, голосовые ассистенты): пара уровней —
   - Сначала быстрая проверка по словарю синонимов (regex). Дёшево и быстро.
   - Если хочется качества — опционально пропускать кандидатов через Lovable AI Gateway (Gemini Flash) с промптом «является ли это запросом услуги из ниши X». Модель проставляет `niche` и `confidence`. Можно включить позже.
3. Если совпадение есть — сохраняем в `leads`: ник автора (`@username`), его user_id, ссылку `t.me/<chat>/<msg_id>`, чат, дату, текст, какой ключевик/ниша сработали.

Дедупликация по `(chat_id, message_id)`. Один автор может встретиться несколько раз — это ок, в UI группируем по нику.

## Схема базы (Lovable Cloud)

- `niches` — id, name (IEEE-ниши).
- `keywords` — id, phrase, niche_id, type (`exact` | `semantic`).
- `chats` — id, tg_chat_id, username, title, niche_id, is_active, joined_at.
- `chat_suggestions` — username, source_chat, mentions_count, status (`new`|`added`|`ignored`).
- `leads` — id, chat_id, message_id, message_link, author_username, author_user_id, author_display_name, message_text, matched_keyword_id, matched_niche_id, posted_at, found_at, is_read, is_starred, notes.
- `worker_state` — last_processed_at, last_error, last_heartbeat (чтобы UI видел, жив ли воркер).

RLS: всё закрыто, читает только ваш авторизованный пользователь. Воркер пишет под service_role и RLS не задевает.

## Веб-интерфейс (минимум)

Четыре экрана:

1. **Лиды** (главный) — таблица: дата, ник (кликабельно → `t.me/<username>`), чат, ниша, сработавший ключевик, превью текста, ссылка на сообщение. Фильтры: по дате (диапазон), нише, чату, ключевику, статусу (прочитан/в избранном). Поиск по тексту. Экспорт в CSV.
2. **Ключевые слова** — список с группировкой по нишам, кнопки добавить/удалить/редактировать. Ваши 7 точных фраз + словари синонимов для 8 ниш предзаполнены.
3. **Чаты** — список активных, кнопка «добавить чат» (вставляешь `@username`), статус подключения. Отдельная вкладка **Suggestions** — что воркер нашёл сам, кнопка «добавить в мониторинг».
4. **Статус воркера** — последний heartbeat, последняя ошибка, обработано сообщений за сутки.

Без авторизации множества пользователей — один логин (вы).

## Воркер на VPS

Один файл `worker.py` (Telethon) + `requirements.txt` + `systemd` юнит для автозапуска. Я подготовлю готовый репозиторий в проекте (папка `worker/`) с:

- подключением к Telegram (sms-логин один раз);
- периодической синхронизацией списка чатов и ключевиков из Lovable Cloud (раз в минуту);
- обработчиком `events.NewMessage` (только новые сообщения, как вы выбрали);
- записью лидов и suggestions в Postgres через `psycopg`;
- heartbeat'ом каждые 30 сек;
- README с командами установки на VPS (Ubuntu): `apt install python3.11`, `pip install -r requirements.txt`, первый запуск, `systemctl enable`.

Конфиг (`.env` на VPS): `TG_API_ID`, `TG_API_HASH`, `TG_PHONE`, `DATABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`.

## Технические детали

- **Стек UI**: TanStack Start (как в проекте) + shadcn + Lovable Cloud. Server functions для чтения/записи под RLS.
- **БД**: Postgres в Lovable Cloud. Миграция создаст все таблицы выше, GRANT'ы и RLS-политики (доступ только аутентифицированному пользователю).
- **Auth**: Lovable Cloud email/password, один аккаунт (вы). Чтобы случайный посетитель preview-ссылки не увидел лиды.
- **Связь воркер ↔ БД**: воркер коннектится напрямую к Postgres по `DATABASE_URL` с service_role-уровнем доступа. Никаких edge-функций между ними не нужно.
- **Realtime**: UI подписывается на таблицу `leads` через Supabase Realtime — новые лиды появляются без перезагрузки.
- **Семантический матчинг через AI**: опциональный флаг в настройках. По умолчанию выключен (regex по словарю), чтобы не тратить токены. При включении — Lovable AI Gateway, Gemini Flash, батчами по 10 сообщений.

## Что я сделаю в этом проекте

1. Включу Lovable Cloud, создам схему БД (таблицы, RLS, GRANT'ы), предзаполню `niches` и `keywords` вашими данными.
2. Соберу веб-интерфейс: 4 экрана, фильтры, экспорт CSV, realtime.
3. Подключу email-логин (один пользователь).
4. Создам папку `worker/` с готовым Python-кодом, `.env.example`, systemd-юнитом и пошаговым README для развёртывания на вашем VPS.
5. Дам вам `DATABASE_URL` и `SERVICE_ROLE_KEY` из Lovable Cloud, чтобы вставить в `.env` воркера.

**Не делаю** автоматически: tgstat API (отложим до решения, нужен ли платный план), AI-семантика (включим переключателем позже, если regex недостаточно).
